# TOGAF Skill - 使用示例

## 示例 1：架构愿景开发

### 场景

需要为数字化转型项目开发架构愿景。

### 输入

"帮我为一个零售银行数字化转型项目开发架构愿景"

### 输出

```markdown
# 零售银行数字化转型 - 架构愿景

## 1. 项目背景

### 1.1 业务驱动力
- 客户期望全渠道无缝体验
- 金融科技竞争压力
- 监管要求日益严格
- 运营成本优化需求

### 1.2 战略目标
- 提升客户数字化体验
- 加速产品创新周期
- 构建开放银行能力
- 实现运营智能化

## 2. 架构愿景

### 2.1 愿景陈述
构建以客户为中心的数字化银行平台，通过开放架构实现生态连接，
借助智能技术优化运营效率，在3年内成为区域领先的数字银行。

### 2.2 目标架构概览

┌─────────────────────────────────────────────────────────────────┐
│                        Digital Customer Experience               │
│    Mobile Banking │ Online Banking │ Open Banking │ AI Assistant │
├─────────────────────────────────────────────────────────────────┤
│                        Digital Core Platform                     │
│  Customer 360 │ Product Factory │ Process Engine │ Risk Engine  │
├─────────────────────────────────────────────────────────────────┤
│                        Integration Layer                         │
│       API Gateway │ Event Bus │ Data Integration │ Security     │
├─────────────────────────────────────────────────────────────────┤
│                        Cloud Infrastructure                      │
│       Hybrid Cloud │ Container Platform │ DevOps │ Observability│
└─────────────────────────────────────────────────────────────────┘

## 3. 利益相关者分析

| 利益相关者 | 关注点 | 期望 |
|-----------|--------|------|
| CEO | 业务增长、竞争力 | 市场份额提升、收入增长 |
| CIO | 技术现代化、成本 | 敏捷IT、降低TCO |
| 业务负责人 | 产品创新、效率 | 快速上市、流程自动化 |
| 客户 | 体验、便捷 | 全渠道、个性化 |
| 监管机构 | 合规、安全 | 数据保护、风险控制 |

## 4. 高层架构原则

1. **客户优先**: 所有设计以提升客户体验为首要目标
2. **开放架构**: 采用标准化API和开放协议
3. **云原生**: 优先采用云原生技术和模式
4. **数据驱动**: 建立统一数据平台，支持智能决策
5. **安全内嵌**: 将安全设计融入架构各层

## 5. 主要工作包

| 工作包 | 范围 | 优先级 | 预期价值 |
|--------|------|--------|---------|
| WP1: 客户360 | 建立统一客户视图 | 高 | 客户洞察、交叉销售 |
| WP2: 新核心银行 | 现代化核心系统 | 高 | 产品灵活性、效率 |
| WP3: 开放银行 | API平台和生态连接 | 中 | 新收入来源 |
| WP4: 智能运营 | 流程自动化和AI | 中 | 成本优化 |
| WP5: 云迁移 | 基础设施现代化 | 高 | 弹性、敏捷 |

## 6. 风险与约束

### 主要风险
- 遗留系统集成复杂度
- 组织变革阻力
- 人才和技能缺口

### 主要约束
- 监管合规要求
- 现有投资保护
- 业务连续性要求
```

## 示例 2：差距分析

### 场景

需要进行应用架构的差距分析。

### 输入

"帮我分析当前应用架构和目标架构的差距"

### 输出

```markdown
# 应用架构差距分析

## 1. 分析范围

本次分析聚焦零售银行核心业务系统的应用架构差距。

## 2. 基线架构概述

### 2.1 当前应用组合

| 应用 | 技术栈 | 状态 | 问题 |
|------|--------|------|------|
| Core Banking | COBOL/IBM z | 老化 | 维护成本高、灵活性差 |
| CRM | Siebel | 过时 | 不支持全渠道 |
| Internet Banking | Java/.NET 混合 | 可用 | 技术债务多 |
| Mobile Banking | Native (2015) | 老化 | 功能有限、体验差 |
| Data Warehouse | Teradata | 可用 | 非实时、成本高 |
| ESB | IBM MQ | 可用 | 单点、不够敏捷 |

### 2.2 当前架构特点
- 单体架构为主
- 紧耦合集成
- 批量数据处理
- 瀑布开发模式

## 3. 目标架构概述

### 3.1 目标应用组合

| 应用 | 技术栈 | 能力 |
|------|--------|------|
| Digital Core | Cloud-native microservices | 产品工厂、账户管理 |
| Customer 360 | Real-time CDP | 统一客户视图 |
| Digital Channels | React/Flutter | 全渠道体验 |
| API Platform | Kong/AWS API GW | 开放银行 |
| Data Platform | Snowflake + Kafka | 实时数据分析 |
| Integration | Event-driven | 松耦合事件驱动 |

### 3.2 目标架构特点
- 微服务架构
- 事件驱动集成
- 实时数据处理
- DevOps CI/CD

## 4. 差距分析矩阵

| 领域 | 当前状态 | 目标状态 | 差距 | 优先级 |
|------|---------|---------|------|--------|
| **核心银行** | 单体/COBOL | 云原生微服务 | 架构重构 | 高 |
| **客户管理** | 孤岛式CRM | Customer 360 | 新建 | 高 |
| **渠道** | 多套系统 | 统一全渠道 | 整合+新建 | 高 |
| **集成** | ESB点对点 | Event-driven | 迁移 | 中 |
| **数据** | 批量ETL | 实时流处理 | 新建 | 中 |
| **API** | 内部SOAP | 开放REST/GraphQL | 新建 | 中 |
| **安全** | 边界安全 | 零信任 | 升级 | 高 |
| **基础设施** | On-premise | 混合云 | 迁移 | 中 |

## 5. 详细差距描述

### 5.1 核心银行系统差距

**当前挑战**:
- COBOL 技能人才流失
- 产品配置周期长（月级）
- 批量处理，无法实时
- 维护成本逐年上升

**目标能力**:
- 产品参数化配置（天级）
- 实时交易处理
- 云原生部署
- 持续交付

**差距弥合策略**:
- 短期: 构建反腐层，API化现有能力
- 中期: 渐进式迁移，领域微服务化
- 长期: 完全替换遗留核心

### 5.2 客户数据差距

**当前挑战**:
- 客户数据分散在多个系统
- 无统一客户标识
- 数据质量参差不齐
- 无实时客户洞察能力

**目标能力**:
- 统一客户主数据
- 实时360度视图
- 客户行为分析
- 个性化推荐引擎

**差距弥合策略**:
- 建立客户主数据管理(MDM)
- 实施客户数据平台(CDP)
- 构建实时分析管道
- 集成ML推荐引擎

## 6. 建议路线图

### Phase 1 (Y1): 基础建设
- 建立API平台
- 实施Customer 360 MVP
- 启动核心系统反腐层
- 云基础设施就绪

### Phase 2 (Y2): 核心现代化
- 核心系统微服务化（账户、支付）
- 全渠道体验重构
- 实时数据平台
- 开放银行API

### Phase 3 (Y3): 深化优化
- 完成核心系统迁移
- AI/ML能力集成
- 全面云原生
- 智能运营

## 7. 关键决策点

| 决策 | 选项 | 建议 | 依据 |
|------|------|------|------|
| 核心系统策略 | 替换/渐进迁移/包装 | 渐进迁移 | 风险可控、投资保护 |
| 云策略 | 公有云/混合云/私有云 | 混合云 | 监管要求、灵活性 |
| 集成模式 | ESB升级/Event-driven | Event-driven | 松耦合、可扩展 |
| 数据平台 | 云数仓/数据湖/Lakehouse | Lakehouse | 统一平台、实时+批量 |
```

## 示例 3：架构原则定义

### 场景

需要为企业架构定义架构原则。

### 输入

"帮我定义一套企业架构原则"

### 输出

```markdown
# 企业架构原则

## 业务原则

### BP-01: 业务连续性
**声明**: 企业运营必须保持连续，不因技术变更或故障而中断。

**依据**:
- 客户服务可用性直接影响业务收入和声誉
- 监管要求保证金融服务的连续性
- 业务中断造成的损失远超过冗余投入

**影响**:
- 所有关键系统需要高可用设计
- 灾备恢复能力需达到业务要求的RPO/RTO
- 变更管理需确保服务不中断
- 需要定期演练业务连续性计划

---

### BP-02: 数据是资产
**声明**: 数据是重要的企业资产，需要按资产进行管理和治理。

**依据**:
- 数据驱动的决策提升业务效能
- 数据质量直接影响运营效率和客户体验
- 监管要求数据的准确性和可追溯性

**影响**:
- 建立企业级数据治理体系
- 定义数据所有权和责任
- 实施数据质量管理
- 数据架构需要企业级规划

---

## 应用原则

### AP-01: 组件化设计
**声明**: 应用系统采用组件化设计，通过标准化接口实现松耦合。

**依据**:
- 组件化提高系统灵活性和可维护性
- 松耦合降低变更影响范围
- 支持能力重用，减少重复建设

**影响**:
- 采用微服务或模块化架构风格
- 定义标准化的服务接口规范
- 组件之间通过API或事件通信
- 避免组件间的直接数据库访问

---

### AP-02: API 优先
**声明**: 所有业务能力通过标准化API暴露，优先设计API再实现功能。

**依据**:
- API是能力重用和生态连接的基础
- API优先设计提高接口质量和一致性
- 支持开放银行战略

**影响**:
- 建立API设计规范和标准
- 实施API全生命周期管理
- 采用API网关统一入口
- API设计需要纳入架构评审

---

### AP-03: 云原生优先
**声明**: 新建应用优先采用云原生架构，充分利用云平台能力。

**依据**:
- 云原生架构提供更好的弹性和敏捷性
- 降低基础设施管理成本
- 支持持续交付和快速创新

**影响**:
- 容器化作为默认部署方式
- 采用托管服务减少运维负担
- 设计无状态应用
- 实施基础设施即代码

---

## 数据原则

### DP-01: 单一事实来源
**声明**: 每个数据实体有且仅有一个权威数据源。

**依据**:
- 消除数据不一致问题
- 简化数据集成和同步
- 明确数据责任归属

**影响**:
- 建立企业数据目录
- 定义主数据管理策略
- 实施数据集成规范
- 禁止建立影子数据副本

---

### DP-02: 数据可访问性
**声明**: 授权用户和系统能够便捷地访问所需数据。

**依据**:
- 数据价值在于被使用
- 支持数据驱动决策
- 提升业务效率

**影响**:
- 建立数据服务和API
- 实施数据自助服务平台
- 定义数据访问权限模型
- 平衡可访问性和安全性

---

## 技术原则

### TP-01: 技术独立性
**声明**: 架构设计尽量独立于特定技术产品，避免供应商锁定。

**依据**:
- 保持技术选型灵活性
- 降低供应商议价风险
- 便于技术演进和迁移

**影响**:
- 采用开放标准和协议
- 设计抽象层隔离具体技术
- 评估供应商锁定风险
- 避免深度使用专有特性

---

### TP-02: 安全内嵌
**声明**: 安全是架构的内在属性，从设计阶段就需要考虑安全。

**依据**:
- 后期补救安全问题成本高
- 监管对安全要求越来越严格
- 安全事件影响企业声誉

**影响**:
- 安全需求纳入架构设计
- 实施安全架构评审
- 采用零信任安全模型
- 安全测试纳入CI/CD流程

---

### TP-03: 可观测性
**声明**: 所有系统必须具备可观测性，支持监控、追踪和分析。

**依据**:
- 快速发现和定位问题
- 支持性能优化和容量规划
- 满足审计和合规要求

**影响**:
- 实施统一的日志、指标、追踪平台
- 应用需要集成可观测性能力
- 定义标准的监控指标
- 建立告警和响应机制

---

## 原则治理

### 应用指南
1. 新项目架构评审必须检查原则符合性
2. 偏离原则需要提交例外申请并获得批准
3. 原则定期（年度）回顾和更新

### 例外处理流程
1. 提交例外申请说明偏离原因
2. 评估偏离的影响和风险
3. 架构委员会审批
4. 记录例外和补偿措施
5. 定期回顾例外状态
```

## 示例 4：构建块定义

### 场景

需要定义一组架构构建块（ABB）和解决方案构建块（SBB）。

### 输入

"帮我定义 API 管理领域的构建块"

### 输出

```markdown
# API 管理领域构建块定义

## ABB: API Gateway

### 基本信息
| 属性 | 值 |
|------|-----|
| ABB ID | ABB-TECH-001 |
| 名称 | API Gateway |
| 类型 | Technology Building Block |
| 层次 | Technology Architecture |

### 功能描述
API Gateway 提供统一的 API 入口点，实现请求路由、认证授权、
流量控制、协议转换等跨切面功能。

### 核心能力
- API 请求路由和转发
- 身份认证和授权
- 流量控制和限流
- 请求/响应转换
- 协议转换（REST, GraphQL, gRPC）
- SSL 终结
- 缓存
- 监控和日志

### 接口规范
- 输入: HTTP/HTTPS 请求
- 输出: 后端服务调用
- 管理接口: REST API

### 质量属性
| 属性 | 要求 |
|------|------|
| 可用性 | 99.99% |
| 延迟 | < 10ms（新增） |
| 吞吐量 | 10,000 TPS |
| 可扩展性 | 水平扩展 |

### SBB 选项

#### SBB-001: Kong Gateway

| 属性 | 值 |
|------|-----|
| 产品 | Kong Gateway |
| 供应商 | Kong Inc. |
| 部署模式 | Self-managed / Kong Cloud |
| 许可 | Open Source / Enterprise |

**优势**:
- 开源版本功能丰富
- 插件生态系统完善
- Kubernetes 原生支持
- 性能优秀

**劣势**:
- 企业版成本较高
- 管理界面需要额外付费

**适用场景**:
- 云原生环境
- 需要高度定制化
- 成本敏感项目

---

#### SBB-002: AWS API Gateway

| 属性 | 值 |
|------|-----|
| 产品 | Amazon API Gateway |
| 供应商 | AWS |
| 部署模式 | Managed Service |
| 许可 | 按使用量付费 |

**优势**:
- 完全托管，无运维
- AWS 生态集成
- 自动扩展
- 按需付费

**劣势**:
- 供应商锁定
- 冷启动延迟（Lambda 集成）
- 高流量成本较高

**适用场景**:
- AWS 优先策略
- 无服务器架构
- 快速上线需求

---

#### SBB-003: Azure API Management

| 属性 | 值 |
|------|-----|
| 产品 | Azure API Management |
| 供应商 | Microsoft |
| 部署模式 | Managed Service |
| 许可 | 分层订阅 |

**优势**:
- 完整的 API 生命周期管理
- 开发者门户内置
- Azure 生态集成
- 混合云支持

**劣势**:
- 供应商锁定
- 学习曲线较陡
- 成本预测较复杂

**适用场景**:
- Azure 优先策略
- 需要完整 API 平台
- 企业级 API 管理

---

### 接口规范

### SBB 选项

#### SBB-004: Kong Developer Portal
- Kong Enterprise 内置功能
- 与 Kong Gateway 深度集成

#### SBB-005: Backstage
- 开源开发者门户框架
- 可扩展插件架构

- SaaS 模式
```
